# 2026-03-20 Phase A-043

## Summary
- completed the `CCATTEST2` Phase 1 same-seed CorePass matrix against the stable private canary host
- confirmed that the non-default wallet path works only when `QR 1 of 2` is scanned with the CorePass in-app QR scanner
- fixed the same-device callback handoff so CorePass no longer opens an in-app browser during the normal callback return path
- recorded the remaining CorePass-side rejection UX limitation as a known external behavior, not a Core Cats app failure

## What This Proves
1. the `sign`-based identify route is no longer the target path for same-seed multi-account rehearsal
2. the current `login`-based identify route is usable for a non-default same-seed wallet only through the CorePass in-app QR scanner
3. device-standard-camera `QR 1 of 2` and same-device mobile handoff still collapse onto the default / active wallet in CorePass for same-seed multi-account cases
4. the browser-side wallet-cap refusal remains correct and happens before any gas-spending commit transaction is prepared

## Known External Limitation
1. when the browser rejects an over-cap request before `QR 2 of 2`, CorePass still shows a generic `Connection unsuccessful` message
2. callback transport variants were tested, including redirect-style returns, `204 No Content`, and `200 text/plain`
3. none of those callback response changes altered the CorePass rejection copy
4. treat that message as a current CorePass-side UX limitation; the authoritative reason remains the browser error state

## Current Launch Consequence
1. same-seed multi-account testers should use the CorePass in-app QR scanner whenever they want a wallet other than the default / active account
2. same-device mobile can still be used for generic single-wallet rehearsal, but it should not be treated as evidence for non-default same-seed wallet selection
3. Phase 1 can be counted as complete because the positive and negative same-seed cases are now reproduced and explained
