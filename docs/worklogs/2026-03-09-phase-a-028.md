# 2026-03-09 Phase A 028 - Pin Foxar remappings for clean clones

Pinned the `foxar/` workspace remappings in-repo so clean clones on Windows do not depend on environment-specific auto-detection to resolve `corezeppelin-contracts`, `spark-std`, and related libraries.

## What changed
1. Added explicit `remappings` to `foxar/foxar.toml` for:
   - `corezeppelin-contracts`
   - `spark-std`
   - `solmate`
   - `openzeppelin-contracts`
   - transitive test utility packages already present in the resolved local Foxar config
2. Declared `test` and `script` paths explicitly alongside `src`, `out`, and `libs` so the workspace is self-contained in a clean clone.

## Why this was needed
1. The active Core Cats contracts and scripts consistently import libraries with the bare-prefix style, for example:
   - `corezeppelin-contracts/utils/cryptography/EDDSA.sol`
   - `spark-std/Script.sol`
2. A clean Windows clone failed to resolve `corezeppelin-contracts/...` because the workspace did not commit the remapping that points to `lib/corezeppelin-contracts/src/`.
3. Changing individual imports to add `src/` would have been a wider and less coherent change because the repo already uses the remapped import style across contracts, scripts, and tests.

## Result
1. `foxar/` no longer relies on machine-specific remapping inference for the Core Cats workspace.
2. The intended operator flow for the `CCATTEST` pilot can proceed from a clean clone after `git pull`, `spark build`, and `spark test`.
